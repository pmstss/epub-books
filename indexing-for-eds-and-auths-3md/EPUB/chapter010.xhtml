Chapter 10

Electronic Files

An indexer’s use of electronic files will depend on the preferences of both the indexer and the publisher, as well as the nature of the particular job. Some indexers, and some publishers as well, still prefer to work from hardcopy proofs. Few if any professional indexers today would consider preparing an index without indexing software. So in a sense, all indexers work with electronic files whether the text is in electronic format or not. Before discussing just how an indexer might use electronic files, a discussion of indexing software and document formats is appropriate.

INDEXING SOFTWARE CAPABILITIES

There are two types of software commonly used for creating indexes. The first is dedicated indexing software, which allows the indexer to create an index as a stand-alone document. The second type includes word processing and page building programs that allow the indexer to embed (or tag) terms within the electronic file and then generate an index using those tags.

Dedicated Indexing Software

Dedicated software is the type most often used by professional indexers. It automates the tasks of alphabetization and checking cross-references, for example, while giving the indexer complete freedom in choosing terms and building the index hierarchy. The three major commercial indexing software programs available at this writing are CINDEX, MACREX, and SKY Index Professional.

Dedicated software is essentially a database-type tool, and the indexer must enter each term into the program. The software provides many time-saving features such as auto-completion of entries, repetition of entries, and warnings for incomplete entries, so the tediousness of typing terms is reduced, and accuracy is enhanced.

When editing an index, the advantages of dedicated software become even more apparent. As mentioned, the software can alphabetize and can do so according to the indexer’s or editor’s specifications. The programs have options for word-by-word or letter-by-letter sorting, with options to force-sort specific terms if necessary. The software can check spelling and verify cross-references quickly and accurately. It even allows individual entry arrays to be viewed so that double-postings can be verified (see Chapter 3).

In addition, the indexer can sort the index by page number and take advantage of many other features such as search and replace capabilities, macros and abbreviations, and dragging and dropping between indexes or between an index and an electronic text document like a Word file or PDF.

Much to an editor’s delight, the indexer can also easily make certain modifications that may be required due to last-minute changes in the text (see Chapter 9 for details).

One of the greatest advantages of dedicated software is the ability to see the index in a fully formatted form as it is being created. This lessens editing time and leaves more time for the indexer to focus on the index structure. The software also can provide line counts and track time spent on a specific index.

What dedicated indexing software will not do is create a concordance or list of terms found in a document. Even if this were possible (and it is with different software), it would not produce an index. No computer program can discern effectively between substantive and passing references, nor can it choose terms for the index that are not actually present in the text. The human element is necessary to produce a quality index.

Once the index is written and edited, the indexer can save the file in a format that can be used by the publisher (an RTF or word processing format, for example).

Embedded Indexing Software

Software that allows the indexer to embed, or tag, terms in the actual text file includes word processing programs, such as Microsoft Word, and page building programs, such as FrameMaker, QuarkXPress, or InDesign. Using these programs, the indexer creates hidden tags or codes that contain terms to be included in the index. After the document is tagged, the software generates a formatted index.

Embedded indexing is often used by technical writers whose end product will be electronic or online documentation. Since page numbers are of no import, indexing can conceivably begin before the final edit is completed.

Waiting to index the document until all writing and editing is complete will probably save time in the long run, however. It will not only avoid re-indexing but could also prevent problems with file-sharing and version control. When the files are ready to be indexed, the indexer should be granted exclusive use of the files.

The great benefit of embedded indexing software is that if page breaks change because of reformatting or because multiple publishing formats are being used, the index tags will be repaginated along with the text, so re-indexing is not necessary. Instead, the index is regenerated after the new format is set.

Problems can arise, however, if text undergoes rewriting or extensive editing and index codes are accidentally moved or deleted. If substantial editorial changes are made, terms may have changed, which would require re-indexing. This becomes a tedious process since changes in embedded indexes are not made easily; each source tag must be edited separately, rather than on a group basis as with dedicated indexing software.

Due to software limitations, it is more difficult to create a quality index by embedding terms than by using dedicated indexing software. Challenges presented by embedded indexes include:

  • No global editing functions – It may not be possible to use a global editing function to edit index source tags; hence, each embedded term will have to be edited individually.
  • Limited sorting options – It may not be possible to force-sort terms or to sort them in page-number order. Some programs can, however, generate a list of tags.
  • No tools for checking cross-references or double-postings.
  • Difficulty in creating page ranges in the index.

Because of these software limitations, some indexers create indexes using dedicated indexing software, and embed the terms in the electronic document. There has been a cry from indexers for better software or software utilities to facilitate embedded indexing. While some utilities exist, they could still be improved to make the indexer’s job easier. Editors should be aware of the difficulties inherent in using embedded indexing software, as more than the usual time may be required to complete and edit an index using this method.

DOCUMENT FORMATS

The format of electronic documents supplied to the indexer will depend on whether the indexer is using dedicated indexing software or is creating an embedded index.

When creating a stand-alone index using dedicated indexing software, text can be provided in PDF files or even in paper form only. Indexers may have preferences in file set-up, such as whether the text is submitted as one file or is separated into individual chapters. Of course, with the appropriate software, the indexer can split or combine files once received, but it saves time if files are created initially in the preferred form. This is another example of when communication between editor and indexer makes a difference.

The size of the files may also determine how they are sent to the indexer. For a scholarly book with very few figures, one PDF may be a manageable size. (When transferring PDFs, there is no need to zip or compress the file, as PDFs are already as compressed as they can get.) For a how-to book with several color illustrations per page, files may have to be broken into chapters before transmission. An alternative is to send the indexer low-resolution files. Editors should ask indexers about email size limitations before sending files.

If the indexer is creating an embedded index, the files will be sent in the format of the software used for writing the document. Therefore, the editor must know that the indexer has the appropriate software—both the type as well as a compatible version.

FILE TRANSFERS

There are several ways in which electronic files can be transferred back and forth between indexer and editor. In addition to attaching files to email, such methods include using FTP and file transfer services.

FTP (File Transfer Protocol)

Transferring large files that may be blocked by an email service can often be handled using FTP, or file transfer protocol. When using FTP, the publisher posts the electronic files to a specific location within its Web site or a special FTP site. The indexer then accesses these files using an FTP program and downloads the files to his computer. All that the indexer needs is the FTP address of the file (which looks like a regular Web address except that it starts with “ftp” instead of “http://”, for example, ftp.companyname.com), as well as the password to access the site.

Some Web browsers, such as Firefox, have FTP plug-ins (small, auxiliary programs) available. You can also find open source (usually free) FTP programs at such sites as Tucows (www.tucows.com) and SourceForge.net (www.sourceforge.net).

The speed of the file transfer will depend, of course, on the speed of the indexer’s Internet connection.

File Transfer Services

There are now a number of online file transfer services for sending large files from one computer to another. These services include SendSpace.com, YouSendIt.com, and TransferBigFiles.com.

Generally, file transfer services use the following process. First, using the service’s Web site, you upload a file to their central server. The file service then sends an email to the file recipient that includes a link to the file. The recipient clicks on the link and the file is automatically downloaded to her computer.

Many of these file transfer services are free. Search the Internet for “file transfer services” to find them.

INDEXERS’ USE OF ELECTRONIC FILES

When working from PDFs, indexers may work only with the electronic file, with a paper printout of the file, or with both. The varied approaches to working with PDFs were discussed at a breakout session during the 2007 annual conference of the American Society of Indexers (ASI). Some panelists said they print the document and work from the paper copy in a traditional manner. Others said they work solely from the electronic file, using widescreen computer monitors or two side-by-side monitors to display both the text and the dedicated indexing software. Some print out a paper copy, but also rely on the ability to search a PDF document as an aid in editing. Actual work methods vary as well, regardless of document format. Some indexers will mark documents, whether paper or electronic, as part of the indexing process; some work without marking.

When creating an embedded index, the indexer must have an appropriate version of the software and be a competent user of the program. The indexer will insert software-specific tags in the document to allow the software to generate an index. Or, indexers may be inserting publisher-defined tags, which are then matched to a stand-alone index, allowing the publisher to do the actual embedding.

Editors and indexers must communicate early during the indexing process about file formats and software versions to ensure successful use of electronic files. In many cases, if format translation is involved, it is helpful to run a test to validate that the indexer’s output can be appropriately translated and used by the publisher. This should be done early in the indexing process to ensure that no problems arise at the time the index is due.
